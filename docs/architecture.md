# 后端架构说明

## 业务边界

一个部署实例可以管理多个 SaaS 应用，但它仍是部署者完全自有的基础设施，不是云平台，也没有外部用户映射：

- `Application` 表示一个 SaaS 产品。
- `Admin` 是部署级后台管理员，可以管理所有应用。
- `User` 是应用内的第一方用户，邮箱和手机号在同一应用内唯一。
- `APIClient` 绑定一个应用，业务后端使用它调用订单和订阅接口。
- `Plan`、`Order`、`Subscription` 都按应用隔离。
- `PaymentConfig` 是部署级全局配置，所有应用共用同一套支付宝和微信商户配置。

管理端应用级接口使用 `X-App-ID` 选择当前应用。公开用户注册和登录使用 `X-App-Key` 标识应用。服务端 API 凭证自身已经绑定应用，因此 `/api/client` 接口不再要求额外传应用 ID。

## Go 目录策略

Go 官方没有规定唯一的项目目录。“最佳实践”应优先保证依赖方向清晰、包职责稳定，而不是照搬大型框架分层。

```text
backend/
├── cmd/api/          # 进程入口和基础设施装配
└── internal/
    ├── app/          # HTTP 组合根、应用管理、计费与支付事务
    └── users/        # 第一方用户、用户认证和后台用户管理
```

当前依赖方向为 `cmd/api -> internal/app -> internal/users`，`users` 不反向依赖 `app`。应用上下文通过 Gin context 传给用户模块，避免循环依赖。

订单支付成功、订单状态变更和订阅生成必须在同一个数据库事务中完成，因此暂时共同保留在 `internal/app`。当计费能力继续扩大时，可以将模型和事务整体迁移到 `internal/billing`，由 `app` 只负责路由组合。

## 数据隔离与一致性

- 应用级管理路由强制校验有效的 `X-App-ID`。
- 用户 JWT 包含 `app_id`，每次请求同时校验用户、应用和状态。
- API Client 鉴权后从凭证取得 `app_id`，订单、套餐和用户查询始终带应用条件。
- 邮箱、手机号、套餐编码和用户订阅使用应用 ID 组成联合唯一索引。
- API Secret 和用户密码只保存 bcrypt 哈希。
- 支付商户配置使用 AES-GCM 密文保存，且微信和支付宝各自全局唯一。
- 支付回调验签后，在事务内锁定订单并幂等生成订阅。
- 有业务数据的应用不能删除，只能停用。

## 初始化

本版本按全新数据库设计，不提供旧单应用数据兼容迁移。首次注册超级管理员时会在同一事务中创建一个“默认应用”。数据库表由 `AUTO_MIGRATE=true` 自动创建，也可以在生产环境中关闭自动迁移并使用导出的结构脚本。
