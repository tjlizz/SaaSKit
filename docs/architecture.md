# 后端架构说明

## 边界

一个部署实例只服务一个 SaaS 产品。系统没有租户、应用归属或平台开发者概念：

- 管理员负责当前实例运维。
- 用户是当前 SaaS 的第一方数据。
- API Client 只是当前业务后端的机器凭证。
- 套餐与支付配置都是实例级数据。
- 订单和订阅直接关联用户 UUID。

## Go 目录策略

Go 官方没有规定唯一的项目目录。“最佳实践”应优先保证依赖方向清晰、包职责稳定，而不是照搬大型框架分层。

```text
backend/
├── cmd/api/          # 组装进程与基础设施
└── internal/
    ├── app/          # HTTP 组合根及强事务计费逻辑
    └── users/        # 第一方用户业务模块
```

当前依赖方向为 `cmd/api -> internal/app -> internal/users`，`users` 不反向依赖 `app`。用户模块拥有用户模型、用户认证、账户接口和后台用户管理。

订单支付成功、订单状态变更和订阅生成必须在一个数据库事务中完成，因此暂时共同保留在 `internal/app`，避免为了形式拆包而引入循环依赖。计费功能继续扩大时，可将这些模型和事务一起迁移到 `internal/billing`，由 `app` 只负责路由组合。

## 不采用的结构

- 不建立全局 `controllers/services/repositories/models` 目录；它会按技术层横切所有业务。
- 不为 GORM 机械创建只有增删改查的 Repository 接口。
- 不把内部代码放入 `pkg`；当前没有供外部 Go 项目复用的公共包。
- 不引入多租户字段或 Application ID。

## 数据一致性

- API Secret 和用户密码只保存 bcrypt 哈希。
- 支付商户配置使用 AES-GCM 密文。
- 支付回调验签后，在数据库事务内锁定订单并幂等生成订阅。
- 有订单的用户禁止删除，只允许停用，以保留财务关联。
- 用户状态在每次用户 JWT 请求时检查，停用立即生效。
